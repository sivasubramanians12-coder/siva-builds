## Order Cascade Monitor — Architecture Pass

Monday heartbeat surfaced the week's Ops domain idea: **Order Cascade Monitor**. Not code yet — a clean concept spec.

The agent's job: watch the Ops OS pipeline (currently Excel-based, transitioning to digital), identify orders that have stalled beyond a configurable threshold, calculate cycle time delta vs. the 17→14 day target, and push a structured morning briefing to Telegram by 7:00 AM IST.

Likely implementation shape: OpenClaw cron (morning, isolated session) + Python script reading from the order data source → simple threshold logic → Telegram message via the `message` tool. No ML needed. The value is in surfacing the stall before a human has to go looking. Core loop: ingest → threshold check → diff since yesterday → notify.

Open questions not yet resolved: (1) what is the exact data source — current Excel sheets or a future DB/API? (2) per-order granularity vs. aggregate daily summary? (3) what threshold triggers a ping — 2 days stalled, 3? These need Siva's operational input before building starts.
